AZ-900-specific vocabulary (no direct 1:1 analog)

Region pair
Two regions in the same geography that Microsoft sequences platform updates and prioritizes disaster recovery between. A DR/resilience concept, distinct from availability zones (which protect within one region).
Sovereign region
A physically and logically isolated instance of Azure for legal/compliance reasons (e.g. Azure Government) — separate from global Azure, not just "more zones."
Resource group
A lifecycle and RBAC/policy container for related resources, inside a subscription. No exact AWS/GCP equivalent — it's a purely organizational construct.
Tenant
Your organization's Microsoft Entra ID directory instance — sits one level above subscriptions, and can contain several of them.
Microsoft Entra ID
The identity directory (users, groups, app registrations). Formerly "Azure Active Directory" / "Azure AD" — the old name is a frequent distractor.
Microsoft Entra Domain Services
A managed, classic AD-compatible directory (domain join, LDAP, Group Policy) for legacy apps — distinct from Entra ID, which is Azure's modern cloud identity directory, not a drop-in AD replacement.
Azure role-based access control (RBAC)
A role assignment = security principal + role definition + scope. Controls who can do what; distinct from Azure Policy, which controls what configuration is allowed.
Resource lock
Blocks delete (CanNotDelete) or delete+edit (ReadOnly) on a resource, regardless of the acting user's RBAC role — even an Owner is blocked until the lock is removed.
Microsoft Purview
Unified data map, catalog, classification, and lineage across your data estate — a governance/compliance tool, not a resource-configuration one.
Azure Arc
Extends Azure's management plane (Policy, RBAC, tags) to servers and Kubernetes clusters running on-premises or in other clouds.
Bicep
A concise authoring language for infrastructure as code that compiles down to standard ARM JSON templates — same deployment engine, friendlier syntax.

Shared responsibility (the one rule that never changes)

Across IaaS, PaaS, and SaaS, the customer always owns data, identities, devices, and access management; Microsoft always owns physical datacenter security. Everything in between (OS, runtime, application) shifts toward Microsoft as you move from IaaS to SaaS.
