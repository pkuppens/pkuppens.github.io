Analytics Workloads on Azure

Tied for the biggest domain. Large-scale analytics (warehouse vs lake, ETL vs ELT), real-time streaming, and Power BI visualization.

The analytics pipeline

Large-scale analytics moves data from sources into an analytical store, then to reports. Two orderings:

ETLELT
OrderExtract → Transform → LoadExtract → Load → Transform
Transform runsIn a staging engine, before loadIn the target store, using its compute
SuitsStructured warehouse loadsBig-data lakes/lakehouses

Analytical data stores

StoreHoldsSchema
Data warehouseCleaned, modelled, structured data (star schema)Schema-on-write
Data lakeRaw data of any structure, cheaply at scaleSchema-on-read
LakehouseLake storage + warehouse-style structure/queriesBoth

Microsoft's big-analytics services: Azure Databricks (an Apache Spark platform for engineering + data science + ML) and Microsoft Fabric (a unified SaaS suite — integration, engineering, warehousing, real-time, and Power BI over one OneLake).

Batch vs streaming

BatchStreaming
InputBounded, collected over timeUnbounded, continuous events
LatencyHigh (scheduled)Low (near real time)
Azure servicePipelines / Fabric, Databricks jobsAzure Stream Analytics, Fabric Real-Time Intelligence
Exam cue "Within seconds of the event" → streaming. "Overnight / on a schedule" → batch.

Power BI — visualize the insight

  • Author in Power BI Desktop; publish to the Power BI service to share.
  • A report is multi-page and detailed; a dashboard is a single page of pinned tiles for at-a-glance monitoring.
  • A data model is tables + relationships + measures (calculated aggregations).
