Relational Data on Azure

Relational fundamentals, the SQL statement families, and the Azure SQL family. The single highest-value fact here: which Azure SQL product fits which scenario.

Relational fundamentals

  • Tables hold rows (records) and columns (fields).
  • A primary key uniquely identifies each row; a foreign key references another table's primary key to enforce a relationship.
  • Normalization structures tables so each fact is stored once — reducing redundancy and update anomalies, at the cost of more joins on read.

SQL statement families

FamilyPurposeStatements
DDLDefine structureCREATE, ALTER, DROP
DMLManipulate dataSELECT, INSERT, UPDATE, DELETE
DCLControl accessGRANT, REVOKE

Common database objects: views (a saved query used like a table), stored procedures (saved executable SQL), and indexes (structures that speed lookups).

The Azure SQL family — pick the right one

ProductModelUse when
Azure SQL DatabasePaaS (single DB / elastic pool)New cloud-native app; least administration
Azure SQL Managed InstancePaaS (instance)Lift-and-shift on-prem SQL Server; needs instance features (SQL Agent, cross-DB queries)
SQL Server on Azure VMsIaaSFull OS/engine control; a feature PaaS lacks
The decision that scores points Managed by Microsoft, brand-new app → SQL Database. Move an existing SQL Server with minimal change → Managed Instance. Need to control the OS / run legacy features → SQL Server on a VM.

Open-source relational services

Azure also offers managed PaaS for the popular open-source engines: Azure Database for MySQL, Azure Database for PostgreSQL, and MariaDB. Choose the one that matches your existing application/engine.

Exam cue "Migrate a PostgreSQL app" → Azure Database for PostgreSQL. "Near-full SQL Server compatibility, managed" → SQL Managed Instance.
