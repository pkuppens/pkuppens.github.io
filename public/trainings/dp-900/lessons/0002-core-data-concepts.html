Core Data Concepts

The vendor-neutral foundation. Know the three kinds of data, the two workload types, the common file formats, and who does what — before any Azure service names appear.

Three ways to represent data

TypeShapeExamples
StructuredFixed schema of rows & columnsA relational Customers table
Semi-structuredSelf-describing tags/keys, flexible schemaJSON, XML, key-value pairs
UnstructuredNo predefined modelImages, audio, video, PDFs, free text
Exam cue "Fixed columns" → structured. "JSON/XML/self-describing" → semi-structured. "Images/video/free text" → unstructured.

Where data lives — file formats & stores

Common file formats split into two camps:

CampFormatsBest for
Row-based / human-readableCSV, TSV, JSON, XMLInterchange, small/medium data, transactional export
Columnar / binaryParquet, Avro, ORCLarge-scale analytics — compression + column pruning

Common data stores include relational databases, document stores, key-value stores, column-family stores, and graph databases. The exam asks you to match a store to a use case (structured & transactional → relational; global JSON → document/Cosmos DB; cheap unstructured → object storage).

Two workloads: OLTP vs OLAP

Transactional (OLTP)Analytical (OLAP)
PurposeRun the businessReport on the business
Access patternMany small reads/writesLarge aggregate reads
DataCurrent, normalized, ACIDHistorical, denormalized (star schema)
ExampleOrder entry, bankingSales trend dashboard
ACID, in one line OLTP systems guarantee Atomicity, Consistency, Isolation, Durability — the reason you trust a relational database with money.

Who does what — the three data roles

RoleOwns
Database administratorAvailability, performance, security, backup/restore of the database
Data engineerBuilding & operating pipelines — ingest, transform, integrate, load
Data analystModelling, exploring, and visualizing data for insight (Power BI)
