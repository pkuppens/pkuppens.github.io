DP-900 · Reference

Glossary

The canonical DP-900 vocabulary. Every lesson uses these names. Where two things are easily confused, the distinction is called out — those pairs are the exam's favourite traps.

Trap pairs DP-900 loves near-miss distractors: OLTP vs. OLAP, ETL vs. ELT, data warehouse vs. data lake, blob vs. file vs. table, batch vs. streaming. Learn each pair as a contrast, not two isolated facts.

Core data concepts D1

Structured data
Data with a fixed schema of rows and columns — the shape a relational table enforces. Example: a Customers table.
Semi-structured data
Self-describing data with tags or keys but no rigid table schema — JSON, XML, key-value pairs. Cosmos DB documents are semi-structured.
Unstructured data
Data with no predefined model — images, audio, video, PDFs, free text. Typically lands in Blob storage.
OLTP (transactional)
Online Transaction Processing: many small, fast reads/writes on current data; normalized; ACID-compliant. Runs the business. Contrast: OLAP.
OLAP (analytical)
Online Analytical Processing: large aggregate queries over historical data; denormalized (star schema); read-optimized. Reports on the business.
ACID
The transaction guarantees: Atomicity, Consistency, Isolation, Durability — the reason relational OLTP systems are trusted for money.
Parquet / Avro / ORC
Columnar/binary file formats optimized for analytics (compression, column pruning). Contrast: CSV/JSON, which are row-oriented and human-readable.
Data roles
Database administrator keeps a database available, performant, secure, backed up; data engineer builds pipelines that move and shape data; data analyst models and visualizes data for insight.

Relational on Azure D2

Normalization
Structuring tables to remove redundancy so each fact is stored once — prevents update anomalies. Trades some read joins for write integrity.
Primary key / foreign key
A primary key uniquely identifies a row; a foreign key references a primary key in another table to enforce a relationship.
SQL statement families
DDL defines structure (CREATE, ALTER, DROP); DML manipulates data (SELECT, INSERT, UPDATE, DELETE); DCL controls access (GRANT, REVOKE).
Database objects
View = a saved query that looks like a table; stored procedure = saved executable SQL; index = a structure that speeds lookups.
Azure SQL Database
Fully managed PaaS relational database (a single database or elastic pool). Microsoft patches and backs it up. The default cloud-native choice. Contrast: SQL Managed Instance, SQL on VM.
Azure SQL Managed Instance
PaaS with near-100% SQL Server engine compatibility (SQL Agent, cross-database queries) — the lift-and-shift target for on-prem SQL Server.
SQL Server on Azure VMs
IaaS: you run full SQL Server on a VM you manage — maximum control, most administration. Use when you need OS-level access or a feature PaaS lacks.
Azure Database for MySQL / PostgreSQL / MariaDB
Managed PaaS for the popular open-source relational engines — pick the one matching your existing app/engine.

Non-relational on Azure D3

Azure Blob storage
Massively scalable object store for unstructured data (files, images, backups, data-lake content). Access tiers: hot, cool, cold, archive — cheaper storage, costlier/slower access as you go colder.
Azure Files
Fully managed file shares over SMB/NFS you can mount like a network drive. Use to lift shared folders to the cloud. Contrast: Blob = objects, Files = a mountable filesystem.
Azure Table storage
A schemaless key-value / NoSQL store for huge volumes of structured, non-relational data, keyed by partition + row key. Cheap and simple.
Azure Cosmos DB
Globally distributed, multi-model NoSQL database with single-digit-millisecond latency and multi-region writes. Use when you need global scale and low latency.
Cosmos DB APIs
Choose by data model / migration source: NoSQL (Core) for documents (native), MongoDB, Cassandra (column-family), Table (key-value), Gremlin (graph).

Analytics on Azure D4

ETL vs. ELT
ETL transforms data before loading (transform in a staging engine); ELT loads raw first, then transforms in the target (lake/warehouse) using its compute. ELT suits big-data lakes.
Data warehouse
A relational analytical store holding cleaned, structured, modelled data (star schema) for BI/reporting.
Data lake
Low-cost storage for raw data of any structure at scale (built on Blob/ADLS). Schema-on-read. Contrast: warehouse = schema-on-write, structured.
Lakehouse
Combines a data lake's cheap raw storage with warehouse-style structure/queries — the model Microsoft Fabric and Databricks promote.
Azure Databricks
An Apache Spark–based analytics platform on Azure for large-scale data engineering, data science, and ML.
Microsoft Fabric
Microsoft's unified, SaaS analytics platform — data integration, engineering, warehousing, real-time intelligence, and Power BI over one OneLake store.
Batch vs. streaming
Batch processes bounded data collected over time, on a schedule (high latency, high throughput); streaming processes unbounded events as they arrive (low latency). Stream Analytics / Fabric Real-Time Intelligence handle streaming.
Power BI
Microsoft's BI/visualization tool: author in Power BI Desktop, publish to the Power BI service, share reports (multi-page, detailed) and dashboards (single-page, pinned tiles).
Power BI data model
The tables, relationships, and measures (calculated aggregations) that a report is built on — the analyst's semantic layer.
